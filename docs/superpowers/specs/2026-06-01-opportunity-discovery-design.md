# Opportunity Discovery Engine Design

## Goal

Add a stage before validation that lets an AI agent mine international public communities and public founder cases into durable startup opportunity candidates. The stage is `discover`, and the lifecycle becomes:

```text
discover -> validate -> business-model -> build -> grow -> operate
```

The engine must preserve every mining run instead of overwriting prior ideas, while still refreshing the latest stage artifacts and the static `playbook/index.html` dashboard.

## Inputs

First version inputs are intentionally file-based so the engine can run without a database:

- community posts from international sources such as Reddit, Hacker News, Indie Hackers, Product Hunt comments, and public forum exports;
- public case records from Indie Hackers, GitHub/open startup repositories, Show HN launches, founder blogs, MicroConf-style public interviews, and other source-linked case summaries;
- optional config for source labels, target markets, thresholds, and output location.

The first implementation can analyze normalized JSON input. Live network fetchers may be added behind the same interface later.

## Candidate Types

The engine produces two candidate families:

- `pain-led`: repeated community pain, tool complaints, manual workflows, alternative seeking, pricing complaints, migration friction, integration friction, or repeated questions.
- `case-led`: public success cases that expose a copyable wedge, acquisition channel, pricing model, validation move, or productized workflow.

The engine should not treat a successful case as permission to clone the product. It records what is copyable and what still requires differentiation.

## Scoring

Each opportunity gets a 0-100 score composed from:

- `painHeat`: repeated pain and discussion strength.
- `caseProof`: public case or revenue/path evidence.
- `reachableChannel`: whether the user surface is named and accessible.
- `soloFounderFeasibility`: whether a solo founder or small AI-native team can validate/build quickly.
- `speedToValidation`: whether the first validation test can run in days.
- `cloneRisk`: penalty when the opportunity is only "copy X" with no wedge.
- `moatDependency`: penalty when success depends on brand, data, network effects, or regulated trust.

The ranking favors executable opportunities over broad markets or viral posts.

## Artifacts

The stage writes:

```text
playbook/stages/discover/input.json
playbook/stages/discover/report.json
playbook/stages/discover/report.md
playbook/stages/discover/handoff.json
playbook/stages/discover/runs/YYYY-MM-DDTHH-mm-ss-<slug>.json
```

The canonical `input/report/handoff` files represent the latest run. The `runs/` directory is append-only for historical idea records.

`report.analysis` includes:

- `radarRun`: run id, run path, source coverage, and candidate counts.
- `signalSummary`: pain signal and case signal counts.
- `opportunityBacklog`: ranked candidates.
- `topCandidates`: highest-priority candidates to validate next.
- `communitySignals`: grouped community evidence.
- `caseSignals`: grouped case evidence.
- `preservation`: where historical runs are stored and how overwrite protection works.

`handoff.summary` passes the top candidate into validation as:

- `candidateId`
- `candidateType`
- `targetUserCandidate`
- `painfulSituation`
- `promiseCandidate`
- `firstReachableChannel`
- `evidenceRefs`
- `recommendedNextExperiment`

## HTML Integration

`skills/startup-playbook-artifacts/templates/index.html` gains a sixth stage tab for `discover`. The page still follows the existing template contract:

- `playbook/index.html` is generated by copying the template shell.
- only the JSON inside `#playbook-data` is replaced;
- template markers, Chart.js, tab navigation, and render functions stay intact.

A renderer script builds dashboard data from `playbook/playbook.json`, `playbook/evidence.json`, and every available stage report. It maps discover analysis into existing display fields plus discover-specific chart fields.

## Tool Boundary

New tool:

```text
tools/opportunity-radar/
  package.json
  README.md
  scripts/run-radar.mjs
  src/radar.mjs
  src/artifacts.mjs
  test/radar.test.mjs
```

Shared renderer:

```text
scripts/render-playbook-index.mjs
scripts/test/render-playbook-index.test.mjs
```

The radar tool owns mining and artifact creation. The renderer owns static HTML generation.

## Safety And Evidence Rules

- Preserve source URLs and short excerpts, not full copyrighted articles.
- Store public-summary-only evidence by default.
- Mark synthetic or fixture data as `mock: true`.
- Separate known evidence, assumptions, and validation gaps.
- Treat case-led candidates as hypotheses, not proof of paid demand.

## Verification

Implementation is complete when:

- unit tests prove pain-led and case-led candidates are generated and scored;
- tests prove historical run paths are unique and not overwritten;
- renderer tests prove `discover` appears in generated dashboard data and HTML while preserving template markers;
- `scripts/validate-startup-artifacts.mjs` accepts the updated protocol/template;
- a sample `discover` run creates `playbook/stages/discover/runs/...json`, latest stage artifacts, and refreshed `playbook/index.html`.
